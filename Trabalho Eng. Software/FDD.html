FDD - Feature Driven Development

HISTORIA

O FDD foi criado em 1997 num grande projeto em java para o United Overseas Bank, em Cingapura.

Nasceu a partir da experiencia de analise e modelagem orientadas por objetos de peter Coad e de gerenciamento de projetos por Jeff de Luca. A metodologia FDD foi aplicada pela primeira vez em 1997, durante um projeto de desenvolvimento de software para um grande banco em Singapura.

O QUE É

É uma metodologia ágil para gerenciamento e desenvolvimento de software.

Ela combina as melhores praticas do gerenciamento ágil de projetos com uma abordagem completa para Engenharia de software orientada por objeto.cc

Foi inicialmente publicada em 1999, no capitulo 6 do livro” Java Modeling in Color with UML”, de Peter Coad, Eric Lefebvre e Jeff de Luca.

A Metodologia foi criada com objetivo de diminuir os custos e tempo utilizados na execução de projetos. Com resultados tangíveis e funcionais.

O FDD é dividido em duas fases:

  • PLANEJAMENTO: Contem 3 etapas:

    DMA – Objetivo de desenvolver o modelo abrangente para o projeto e tem como resultado a confecção dos diagramas de classes e em sequencia a modelagem dos objetos.

    CLF – Tem como objetivo construir a lista de funcionalidades do sistema.

    PPF – Ela consiste fazer o planejamento com funcionalidades e nela que se determina a sequencia de desenvolvimento levando a consideração as dependências entre as funcionalidades e em classes envolvidas. Tambem nesta etapa os programadores lideres recebem os cronogramas e atividades definidas.

  • CONSTRUÇÃO: Cotem 2 etapas:

    DPF – Nele são detalhados os requisitos para funcionalidade, também são construído esqueletos de cada classe.

    CPF – Ela tem como objetivo a codificação, ou seja o preenchimento de cada esqueleto e código. Nesta etapa também, acontece os testes.

  • PRATICAS DO FDD

    Modelagem dos objetos de domínio;

    Construção de diagramas de classes UML (Unified Modeling Language) que descrevem os objetos relevantes dentro do domínio do problema, bem como os relacionamentos entre eles. Para complementar os diagramas de classe UML, são desenvolvidos diagramas de sequencia UML que descrevem explicitamente como os objetos interagem para cumprir suas

  • Desenvolvendo através de funcionalidades;

    É feita a identificação das funcionalidades do sistemas (definidas pelo cliente). Após isso, inicia-se o projeto e a construção de cada uma delas. Uma vez identificadas as funcionalidades serão utilizadas para guiar o desenvolvimento no FDD, tendo como objetivo mostrar o progresso através da implementação das mesmas.

    A execução das funcionalidades, ou conjunto delas, não deve exercer de duas semanas.

  • Propriedade individual das classes;

    Cada classe ou conjunto de classes é de responsabilidade do individuo. Isso pode ser uma vantagem e uma grande desvantagem em alguns pontos de vista, pois a saída do programador proprietário da classe pode gerar perda de tempo no projeto.

  • Equipes de funcionalidades;

    A prática da propriedade individual da classe atribui classes a desenvolvedores específicos. Contudo sabe se que o desenvolvimento deve ser por funcionalidade. Por isso, são definidas equipes, com seus respectivos desenvolvedores líderes, onde os componentes possuem as propriedades das classes, e é atribuído a eles um conjunto de funcionalidades. A equipe de funcionalidades deve ter no mínimo 3 e no máximo 6 programadores envolvidos.

  • Inspeções

    Devem ser feitas durante e ao final de cada iteração, para assegurar a qualidade do projeto e do código. O objetivo principal das inspeções é a detecção de defeitos. É uma ferramenta para eliminação de erros e uma grande oportunidade de aprendizado.

  • Construções regulares;

    Devem ocorrer durante as iterações, na execução e um conjunto de funcionalidades, para detectar, prematuramente, erros de integração. Uma construção regular assegura também que haja sempre um sistema atual e executável para ser apresentado ao cliente.

  • Administração de configuração;

    Utilização de um sistema de um controle de versões para datar e manter um histórico das alterações feitas em cada classe. Bem como, no que se refere aos requisitos, análise e projeto de modo que facilite a visualização das modificações feitas.

  • Relatórios de resultados;

    O FDD sugere que todos os resultados ocorridos durante o projeto sejam disseminados para todos os membros da equipe e clientes.

FASES DO FDD

Vantagens do FDD

  • Recomendado para qualquer tipo de desenvolvimento.

  • Foco em “características de valor para o cliente”

  • FDD prioriza aquilo que o cliente prioriza.

  • FDD possui requisitos mais formais.

  • Seus princípios e práticas proporcionam um equilibro entre as filosofias tradicionais e as mais extremas, proporcionando uma transição mais suave para organizações mais conservadoras;

Desvantagens

  • Ainda existem questionamento sobre a eficácia / aplicabilidade sobre o tamanho mini de um time FDD.

  • Controvérsias sobre o tamanho mínimo de um time FDD;

    • algum problema, essa reunião não tem o propósito de pensar em soluções.

Considerações finais

É um método ágil e altamente adaptativo.

Oferece vantagens dos métodos pesados.

Oferece vantagens dos métodos extremamente ágeis.

É orientada as necessidades dos clientes, gerentes e desenvolvedores.
